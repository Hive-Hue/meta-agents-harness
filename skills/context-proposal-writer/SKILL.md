# Context Proposal Writer

Use this skill to improve the readability and utility of proposals generated by `mah context propose --from-session`.

## Goal

Transform raw proposal text into concise, operationally useful writing for human reviewers.

## Input

- Session-derived proposal object with:
  - `summary`
  - `rationale`
  - `proposed_content`
  - `proposed_frontmatter`
  - `source_ref`

## Output Rules

- Keep proposal intent unchanged.
- Do not invent facts, tools, systems, capabilities, or outcomes.
- Keep governance model unchanged:
  - draft remains draft
  - no auto-promotion language
  - preserve explicit review requirement
- Improve:
  - summary clarity (single clear sentence)
  - rationale intelligibility (organized bullets/sections)
  - proposed content structure (actionable headings and steps)
- Keep operational density and technical tone.
- Avoid marketing language.

## Content Constraints

- Preserve source provenance and references.
- Keep safety caveat about manual review.
- Prefer short, explicit statements over generic phrasing.
- If confidence is low, keep text conservative and factual.

## Suggested Rewrite Shape

1. `summary`: one-line statement of what the proposal captures and why it matters.
2. `rationale`: short sections:
   - Session Context
   - Observed Signals
   - Why This Memory Is Reusable
3. `proposed_content`:
   - Task Breakdown
   - Solution Summary (required): concise implementation recap with concrete code snippets, commands executed, and key technical decisions
   - Pitfalls / failure notes
   - Validation checklist

## Reusability Requirement

- In `proposed_content`, always include a `## Solution Summary` section in addition to `## Task Breakdown`.
- `Solution Summary` must include:
  - Relevant code excerpts or pseudo-snippets tied to the actual change.
  - Practical commands used for implementation/validation.
  - Key points/decisions needed to reproduce or extend the solution later.
- Keep it factual and source-grounded; do not add details not present in session evidence.

## Non-Goals

- Do not change routing or expertise decisions.
- Do not alter proposal IDs or governance states.
- Do not add unsupported claims.
