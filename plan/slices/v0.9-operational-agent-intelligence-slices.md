# v0.9 Operational Agent Intelligence - Executable Slices

**Version:** 0.9.0
**Status:** S1-S7 delivered
**Audit status:** S1-S7 delivered (2026-04-23); S8 deferred to v0.9.x
**Input RFC:** `plan/v0.9-operational-agent-intelligence-rfc.md`
**Input Spec:** `specs/meta-agents-harness-v0.9-operational-agent-intelligence-spec.md`

---

## 1. Overview

This document breaks `v0.9` into executable implementation slices.
The purpose is to make the release shippable in bounded increments while preserving MAH's identity:

- CLI-first
- runtime-agnostic
- policy- and topology-bounded
- lighter than a managed-agents platform

The target release story remains:

```text
route the right agent
load the right context
show the work happening
```

---

## 2. Release Outcomes

`v0.9` is considered complete when MAH can support, in a coherent operator flow:

1. explainable expertise routing
2. Context Manager as a visible subsystem
3. lightweight lifecycle visibility for runs/sessions
4. evidence-backed compounding via expertise/context flows
5. bootstrap messaging that communicates the value story

This plan intentionally excludes:

- board UX
- web app requirements
- daemon-first architecture
- company/workforce simulation
- heavy governance product surfaces

---

## 3. Slice Strategy

The slices are ordered to maximize product coherence early:

1. improve routing outputs first
2. align Context Manager naming and operator surfaces
3. add lifecycle persistence primitives
4. expose lifecycle through `mah sessions status`
5. connect lifecycle into `run --trace` and delegated execution
6. strengthen compounding and bootstrap narrative

The first shippable internal milestone is after Slice 3.
The first user-facing coherent beta is after Slice 5.

## Slice S1 - Expertise Operator UX

**Current status:** delivered

**Already delivered**

- `mah expertise recommend --task "..."` exists and produces concise ≤5-line text summary by default.

**Still missing**

- none

Make expertise routing feel like a release feature rather than an internal capability.

**Operator outcome**

The operator can answer "why this agent?" from default CLI output without reading JSON.

**In scope**

- tighten default text output of `mah expertise recommend`
- tighten default text output of `mah expertise explain`
- add optional `--agent <agent>` support to `mah expertise explain`
- ensure text output includes:
  - recommended/inspected agent
  - confidence
  - short capability-fit explanation
  - short evidence summary
  - relevant constraints/penalties

- `docs/expertise-model-foundation.md`
- `README.md`

- `mah expertise recommend --task "..."` returns useful output in <= 5 lines by default
- `mah expertise explain --task "..."` returns ranked decision rationale in a concise text mode

- `--json` output remains stable for automation

**Test plan**

- unit tests for new formatter helpers
- integration tests for:
  - recommend default text output
  - explain default text output
  - explain agent-scoped flow
- non-regression on existing JSON payloads

**Out of scope**

## Slice S3 - Lifecycle Event Foundation

**Current status:** delivered

**Already delivered**

- session/provenance infrastructure exists and can persist session-related metadata.
- `types/lifecycle-event-types.mjs` — canonical `LIFECYCLE_STATES` and `getCurrentState()`
- `scripts/m3-ops.mjs` — `recordLifecycleEvent()` and `getLifecycleEvents()` for file-based persistence
- `runDelegate` records `queued → routed → completed/failed` lifecycle events
- `dispatchHeadless` records `running → completed/failed` lifecycle events
- Events persisted to `.mah/sessions/lifecycle-events/<session-id>.json`

**Still missing**

- none

**Goal**

Introduce a minimal lifecycle event model without adding a daemon or separate event system.

**Operator outcome**

Every run can be understood as moving through a bounded lifecycle.

**In scope**

- define canonical lifecycle states:
  - `queued`
  - `routed`
  - `context_loaded`
  - `running`
  - `blocked`
  - `completed`
  - `failed`
- add lifecycle event recording to existing session/provenance infrastructure
- include routing and context summaries where available
- ensure blocked/failure events carry structured reasons

**Likely files**

- `scripts/meta-agents-harness.mjs`
- `scripts/m3-ops.mjs`
- any provenance/session helpers used by `run` and `delegate`
- `docs/sessions-interop.md`

**Acceptance criteria**

- lifecycle events are persisted for a run without requiring a new daemon
- `routed` events can include selected agent and routing summary
- `context_loaded` events can include a count/summary of retrieved context
- `blocked`, `completed`, and `failed` events include a stable reason/result shape

**Test plan**

- unit tests for lifecycle event schema/serialization
- integration tests for run flows with persisted event sequence
- non-regression tests for existing provenance/session behavior

**Out of scope**

- streaming UI
- websocket transport
- background polling workers

## 6. Recommended Implementation Order

### Phase A - Make the product legible

1. Slice S1
2. Slice S2

**Reason**

This makes the existing system easier to understand immediately and yields the fastest operator-facing value.

### Phase B - Make execution visible

3. Slice S3
4. Slice S4
5. Slice S5

**Reason**

This creates the lightweight lifecycle story that differentiates MAH from "just a runtime wrapper" without requiring board or daemon architecture.

### Phase C - Make the improvement loop explicit

6. Slice S6
7. Slice S7

**Reason**

This turns the release into a coherent narrative from first bootstrap through repeated use.

### Phase D - Optional bounded expansion

8. Slice S8

**Reason**

Useful, but not required to ship the core `v0.9` story.

## 8. Exit Criteria for `v0.9.0`

`v0.9.0` is ready when:

- S1-S5 are complete and stable
- S6-S7 are complete in docs/help/output form
- default operator flows are coherent:
  - route the task
  - explain the routing
  - load context
  - inspect session state
  - see the execution timeline
  - understand how successful runs improve future runs

`v0.9.0` does not require:

- board UI
- daemon architecture
- goal/budget/governance extras beyond bounded MVP signals

---

## 9. Cut Strategy

If the release needs to be narrowed, cut in this order:

1. Slice S8
2. advanced enhancements inside Slice S6
3. non-essential copy/help refinements inside Slice S7

Do **not** cut first:

- S1
- S3
- S4

If those are cut, the release loses the core `v0.9` identity.

---

## 10. Final Recommendation

The strongest implementation path is:

- ship the operator clarity slices first
- then ship lifecycle visibility
- then connect the compounding and bootstrap narrative

That yields a release that is:

- more sellable than current MAH
- still recognizably MAH
- materially different from both Multica and Paperclip
- achievable without a platform rewrite
