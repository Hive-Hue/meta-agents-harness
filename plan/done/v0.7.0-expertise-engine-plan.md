## Contexto

Baseado na spec: `specs/meta-agents-harness-expertise-model-spec.md`.

Objetivo do release `v0.7.0`: evoluir o **Expertise Model** de artefato estático para **camada operacional de inteligência** que influencia delegação, confiança, roteamento e governança.

Princípio norteador:
- Topologia e policy definem o que é permitido.
- Expertise define quem é melhor candidato entre os alvos permitidos.

## Outcome de Produto (v0.7.0)

Ao final do release, MAH deve suportar:
1. Expertise estruturada como objeto de primeira classe (não blob textual).
2. Delegação expertise-aware com scoring explicável.
3. Sinais de confiança/validação (`declared/observed/validated`) com efeitos no roteamento.
4. Registro consultável de expertises e visão básica de métricas.
5. Base técnica para export/federação em releases seguintes (sem big-bang).

## Escopo do v0.7.0 (In)

- Modelo de dados estruturado para expertise.
- Persistência local versionada no repositório.
- Leitura do modelo por runtime/orquestrador.
- Roteamento por expertise com fallback seguro.
- Coleta de evidências mínimas de execução.
- Cálculo inicial de confidence e validation status.
- CLI de inspeção e explainability.
- Testes de contrato, integração e regressão.

## Fora do Escopo (Out)

- Ontologia completa de domínios.
- Federação full cross-org com handshake de confiança completo.
- Auto-promoção de expertise para níveis críticos sem policy explícita.
- UI web dedicada (entregaremos CLI e payloads estruturados).

## Arquitetura Alvo (v0.7.0)

## Camadas

1. `Expertise Catalog` (declared)
- Fonte canônica de expertises por agente/time.

2. `Evidence Store` (observed)
- Sinais de execução, review e custo/latência.

3. `Trust Layer` (validated/restricted/revoked)
- Estado de validação e confidence para decisão operacional.

4. `Routing Engine`
- Scoring de candidatos permitidos por topologia/policy.

5. `Explainability`
- Motivos de seleção e fallback em formato auditável.

## Milestones

## M1 — Structured Expertise Foundation

Entrega:
- Tipos e schema do modelo de expertise.
- Validador (`validate:expertise`) com erros/warnings.
- Loader/normalizer para catalog e registry.
- Convenção de arquivos por owner (`agent`/`team`).

Critérios de aceite:
- Expertise inválida bloqueia validação.
- `mah validate:expertise --json` com shape estável.
- Contract tests do schema passando.

## M2 — Expertise-Aware Routing (MVP)

Entrega:
- `scripts/expertise-routing.mjs` com algoritmo de seleção:
  1. policy/topology allowed set
  2. environment compatibility
  3. trust tier requirement
  4. expertise match score
  5. confidence/validation adjustment
  6. fallback/escalation
- Integração com `delegate` e `mah_delegate_agent`.
- `mah explain delegate --trace` com decisão explicável.

Critérios de aceite:
- Delegação nunca viola policy/topologia.
- Quando há múltiplos candidatos permitidos, expertise altera ranking.
- Explain inclui score, filtros aplicados e razão do fallback.

## M3 — Observed Evidence + Confidence

Entrega:
- Coleta de sinais de execução (sucesso/falha, review pass, latência, custo).
- Agregador de métricas por expertise.
- Confidence inicial (bands + score numérico opcional).
- Lifecycle básico: `draft|active|restricted|experimental|deprecated`.

Critérios de aceite:
- Evidência é gravada por run em formato auditável.
- Alterações de confidence impactam ranking de roteamento.
- Estados `restricted/revoked` bloqueiam uso onde aplicável.

## M4 — Registry + Operator UX (CLI)

Entrega:
- Comandos:
  - `mah expertise list`
  - `mah expertise show <id>`
  - `mah expertise recommend --task "<...>"`
  - `mah expertise evidence <id>`
- Saída `--json` para automação.
- Explain panel textual no CLI para diagnóstico operacional.

Critérios de aceite:
- Operador consegue responder: “quem é melhor para essa tarefa e por quê”.
- Registry mostra status, confidence, owner, ambiente, restrições.
- Comandos cobertos por testes de integração.

## M5 — Exportable Expertise Contract (bounded) ✅ COMPLETO

Entrega:

## Algoritmo de Roteamento (v0.7.0 MVP)

```text
Input: task, source_agent, candidate_targets (já policy/topology-allowed)

1) Filtrar por allowed_environments
2) Filtrar por trust_tier_required
3) Filtrar por validation_status bloqueante (restricted/revoked)
4) Calcular match_score por capabilities/domain/input_contract
5) Ajustar por confidence_score e evidence freshness
6) Aplicar penalidades (escalation_rate/failure_modes)
7) Ordenar candidatos e escolher top-1
8) Se score < threshold => escalate/approval path
9) Emitir explain payload
```

## Data Contracts e Compatibilidade

## Versão
- `expertise.schema_version: "v1"`

## Compatibilidade
- `v1` obrigatória em `v0.7.0`.
- Campos desconhecidos: warning (não erro), salvo quando `--strict`.

## Segurança
- Nenhum campo de expertise concede permissão por si só.
- Expertise apenas ranqueia dentro do conjunto permitido por policy/topologia.

## Métricas de Sucesso do Release

1. Qualidade de delegação:
- redução de escalations indevidas.
- aumento de review pass rate em tarefas roteadas por expertise.

2. Explainability:
- 100% de decisões de `delegate` com payload explicável em `--trace`.

3. Operabilidade:
- `mah expertise *` cobrindo descoberta, inspeção, recomendação e evidência.

4. Segurança:
- zero bypass de policy/topologia por causa de expertise.

## Riscos e Mitigações

1. Scoring ruim no início
- Mitigação: thresholds conservadores + fallback para roteamento atual.

2. Overfitting por baixa amostra
- Mitigação: confidence cap até mínimo de evidências.

3. Complexidade operacional
- Mitigação: CLI simples + defaults fortes + explain claro.

4. Mistura de policy com expertise
- Mitigação: enforcement explícito “policy first”.

## Dependências e Decisões Abertas (bloqueadores)

1. Ownership principal do expertise:
- agente, time ou ambos.

2. Fórmula de confidence:
- heurística inicial vs score calibrado.

3. Fonte de verdade para validation status:
- policy manual, review humana, avaliação automática, ou híbrido.

4. Nível de export em `v0.7.0`:
- apenas contrato local ou já cross-team habilitado.
