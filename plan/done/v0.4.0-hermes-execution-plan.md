# Plano de Execução — `v0.4.0` Hermes Runtime

## Status

- execução do plano: [done]
- resultado entregue: suporte Hermes entrou no release `v0.4.0`
- observação:
  - a parte de projeção/artefatos Hermes foi concluída dentro do boundary permitido pelo plano, com suporte bounded e documentação explícita

## Objetivo

Executar a especificação `specs/mah-v0.4.0-hermes-runtime-spec.md` de forma incremental, testável e compatível com a tese do produto:

- Hermes entra como runtime de primeira classe
- MAH continua runtime-agnostic
- o release não vira fork de Hermes
- mudanças continuam cabendo em uma minor disciplinada

## Tema do release

**Deep Hermes runtime support with selective absorption**

Tradução prática para execução:

- aprofundar suporte operacional do runtime Hermes
- absorver apenas patterns úteis e de baixo risco
- preservar `meta-agents.yaml` como fonte canônica
- não puxar para `v0.4.0` o escopo de remote execution foundation, federation, policy engine ou confidential execution

## Regras de escopo

### Entra em `v0.4.0`

- detecção, seleção e forcing de Hermes
- adapter Hermes no mesmo modelo dos demais runtimes
- mapeamento de comandos Hermes para o command surface do `mah`
- explainability/diagnostics Hermes-aware
- suporte Hermes em validação e contrato de runtime
- projeção/sync de artefatos Hermes se houver formato mínimo bem definido
- docs e testes compatíveis com o novo runtime
- metadata Hermes limitada e additive, se necessária

### Não entra em `v0.4.0`

- feature parity completa com Hermes
- remote execution foundation multi-runtime
- federation/interconnect
- confidential execution
- policy engine
- expertise operationalization profunda
- reorientar o produto inteiro em torno de Hermes

## Critérios de sucesso do release

- Hermes aparece como runtime suportado real em `detect`, `doctor`, `validate:*`, `explain`, `run`, `list:crews`, `use`, `clear`
- o adapter Hermes passa no contrato mínimo de runtime
- qualquer limitação de Hermes é surfaced com erro honesto e explicável
- a configuração canônica continua centrada em `meta-agents.yaml`
- o changelog consegue resumir o release em 3–5 bullets claros
- smoke/contract/diagnostics/validate continuam verdes

## Estratégia de execução

## Fase 0 — Discovery técnico mínimo

### Objetivo

Fechar os unknowns de Hermes antes de tocar o fluxo principal.

### Entregáveis

- matriz mínima de comandos Hermes:
  - binário direto
  - wrapper desejado
  - comandos equivalentes a `list:crews`, `use`, `clear`, `run`, `doctor`, `check:runtime`
- entendimento das semânticas de sessão:
  - novo contexto
  - continuar sessão
  - session id
  - working/session root
  - espelhamento de sessão, se existir
- entendimento do formato de configuração/artifacts Hermes, se houver projeção viável

### Saída esperada

- decisão explícita de:
  - quais comandos terão parity direta
  - quais comandos terão suporte parcial
  - quais ficarão como unsupported-but-explicit

### Gate para seguir

- Hermes consegue ser encaixado no adapter model sem refactor estrutural do dispatcher

## Fase 1 — Runtime foundation no adapter model

### Objetivo

Adicionar Hermes como runtime de primeira classe no mesmo contrato dos demais.

### Mudanças

- incluir `hermes` em `RUNTIME_ORDER`
- adicionar entrada `hermes` em `RUNTIME_ADAPTERS`
- definir:
  - `markerDir`
  - `wrapper`
  - `directCli`
  - `capabilities`
  - `commands`
- manter o contrato mínimo intacto

### Critérios

- `test:contract` continua verde
- `mah detect` encontra Hermes quando forçado ou quando marker/CLI estiver presente
- `mah --runtime hermes detect` funciona

### Possíveis issues

- `feat(runtime): add hermes adapter skeleton`
- `feat(runtime): add hermes runtime detection and command mapping`

## Fase 2 — Command surface Hermes-aware

### Objetivo

Integrar Hermes aos comandos centrais do MAH.

### Comandos-alvo

- `mah detect`
- `mah doctor`
- `mah check:runtime`
- `mah validate:runtime`
- `mah validate`
- `mah explain`
- `mah run`
- `mah list:crews`
- `mah use`
- `mah clear`

### Execução

- mapear command plans Hermes no adapter
- ajustar normalização de args em `run`
- emitir warnings claros para semânticas específicas de Hermes
- garantir que explainability mostre:
  - runtime resolvido
  - exec
  - args
  - warnings
  - limitações

### Gate

- nenhum branch ad hoc Hermes-specific espalhado pelo dispatcher
- tudo passa por adapter/capabilities e explainability consistente

## Fase 3 — Compatibilidade com `meta-agents.yaml`

### Objetivo

Adicionar suporte Hermes sem criar “um segundo universo de config”.

### Mudanças permitidas

- campo runtime `hermes` sob metadata/additive config
- capability metadata additive e opcional
- runtime overrides Hermes apenas se realmente necessários

### Mudanças proibidas

- reestruturação completa do schema canônico
- configs Hermes-only dominando o modelo global

### Critérios

- Hermes cabe no mesmo modelo declarativo dos outros runtimes
- extensões são additive e de baixo risco

### Possíveis issues

- `feat(config): additive hermes runtime metadata`
- `feat(config): bounded hermes runtime overrides`

## Fase 4 — Projeção e artefatos Hermes

### Objetivo

Suportar Hermes no pipeline `canonical config -> runtime-specific artifacts`, mas apenas se o formato mínimo for claro e determinístico.

### Opções

#### Opção A — projeção completa mínima

- gerar arquivos Hermes equivalentes aos existentes para outros runtimes
- incluir drift-check em `sync --check`

#### Opção B — suporte parcial explícito

- gerar apenas o subconjunto viável
- deixar claro no sync/report o que ainda não é projetado

### Critério de decisão

Se a projeção Hermes exigir inventar abstrações frágeis ou não determinísticas, reduzir escopo e marcar como partial support.

### Critérios

- qualquer artefato Hermes gerado precisa ser:
  - determinístico
  - diffável
  - validável

## Fase 5 — Diagnostics, explainability e honestidade operacional

### Objetivo

Garantir que Hermes participe do envelope diagnóstico do produto.

### Entregas

- Hermes dentro de `mah.diagnostics.v1` onde aplicável
- `mah explain` com Hermes resolution plan
- `doctor` e `validate:*` mostrando limitações e warnings honestos

### Casos importantes

- CLI Hermes ausente
- marker presente mas wrapper/binário ausente
- comando suportado parcialmente
- semântica de sessão não equivalente

### Gate

- nenhum caso relevante falha silenciosamente

## Fase 6 — Testes e CI

### Objetivo

Manter release shippable e prevenir regressões cross-runtime.

### Matriz mínima de testes

#### Smoke

- detect com Hermes forçado
- help contendo Hermes quando aplicável
- comandos centrais retornando erro honesto ou sucesso esperado

#### Contract

- adapter Hermes cumpre o contrato mínimo

#### Diagnostics

- envelopes `--json` continuam estáveis com Hermes
- explain/doctor/validate refletem Hermes corretamente

#### Sync/config

- `validate:config`
- `validate:sync`
- `check:meta-sync`

### CI

- expandir validação para Hermes-aware adapter/config logic
- não exigir CI com Hermes fully installed se isso ainda não for viável
- cobrir ao menos:
  - contract
  - config compatibility
  - diagnostics behavior

## Fase 7 — Selective absorption

### Objetivo

Absorver patterns úteis de Hermes sem captura de arquitetura.

### O que pode entrar

- metadata de persistence capability
- hints de backend support
- notes de skill packaging compatível
- gateway awareness como metadata futura, não feature ativa

### O que não pode entrar

- lifecycle Hermes como lifecycle global do produto
- automações Hermes como surface dominante
- visão “always-on assistant platform” dentro de `v0.4.0`

### Regra operacional

Só absorver o que puder ser expresso como:

- capability metadata
- adapter behavior
- generated artifact opcional
- doc de compatibilidade

## Dependências e decisões que precisam ser fechadas cedo

- nome do wrapper Hermes:
  - `hermesh`
  - outro nome oficial
- diretório marker/config root:
  - `.hermes`
  - outra convenção real
- comandos equivalentes reais de:
  - `use`
  - `clear`
  - `list:crews`
  - `doctor`
  - `check:runtime`
- formato de artefato Hermes que seja realmente projetável
- semântica de sessão mínima suportável sem quebrar normalização dos demais runtimes

## Critérios de corte de escopo

Se algum item:

- exigir refactor amplo do dispatcher
- exigir redesign do `meta-agents.yaml`
- exigir remote execution foundation
- puxar policy/federation/confidential
- introduzir semântica Hermes que não consegue ser abstraída com honestidade

então esse item sai de `v0.4.0`.
