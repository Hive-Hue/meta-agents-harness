# RFC - `v0.9` Operational Agent Intelligence

## Status

- status: [proposed]
- owner: core MAH
- input spec: `specs/meta-agents-harness-v0.9-operational-agent-intelligence-spec.md`
- objective:
  - turn the `v0.9` proposal into a more decision-oriented RFC focused on scope, CLI contracts, rollout, and acceptance criteria
  - preserve MAH's identity as an orchestration intelligence layer for runtimes, without drifting into a managed agents platform

---

## 1. Proposed Decision

Meta Agents Harness `v0.9` should ship under the thesis:

**Operational Agent Intelligence**

Core message:

- MAH chooses who should execute based on expertise within policy/topology bounds
- MAH loads bounded operational context for the selected agent
- MAH makes execution state visible through a lightweight, CLI-first lifecycle

Primary decision:

- **approve `v0.9` as a productization release**
- **do not approve `v0.9` as an expansion into a board/app/daemon platform**

---

## 2. Motivation

MAH already has the right technical pillars:

- `mah expertise recommend` and `mah expertise explain`
- `mah context find`, `mah context explain`, `mah context propose`
- `mah run --with-context-memory`
- `mah explain`, `mah sessions`, `mah graph`, `trace`, and `audit/provenance`

The current problem is not lack of foundation.
The problem is lack of narrative cohesion and operator experience.

Today the product can still feel like:

- a CLI for runtime unification
- a collection of technical subsystems

When it could be perceived as:

- an operational intelligence layer for agents and runtimes

---

## 3. Release Goal

By the end of `v0.9`, an operator should be able to answer, from the CLI alone:

1. **Who should do this task?**
2. **Why was this agent selected?**
3. **What operational context was loaded for execution?**
4. **What state is the execution in right now?**
5. **What did this session teach the system for later improvement?**

---

## 4. Product Boundaries

### What `v0.9` is

- an evolution of MAH from unified runtime control to orchestration intelligence
- a CLI-first release
- a lightweight layer of operational visibility
- a release that promotes expertise, context, and lifecycle to first-class features

### What `v0.9` is not

- a managed agents platform in the Multica sense
- an autonomous company system in the Paperclip sense
- a Kanban board
- a mandatory web app
- a persistent daemon/control plane as a core requirement

Decision:

- every `v0.9` feature must fit inside the current MAH model: **canonical config + runtime adapters + CLI + local artifacts + provenance**

## 6. Scope Decisions

## D1 - Explainable routing becomes a core feature

**Decision:** promote `mah expertise recommend/explain` from a technical capability to a release feature.

### Implications

- default output must be short
- routing must answer "why this agent?" without requiring `--json`
- the explanation must mention:
  - capability fit
  - confidence
  - evidence summary
  - policy/topology fit

### Minimum contract

```bash
mah expertise recommend --task "..."
mah expertise explain --task "..."
mah expertise explain --agent <agent> --task "..."
```

### Acceptance criterion

- the operator can understand the selection in up to 5 lines of default output

---

## D2 - Context Manager becomes the public name

**Decision:** keep `mah context` as the CLI namespace, but refer to the subsystem publicly as **Context Manager**.

### Implications

- docs and changelog must use "Context Manager"
- the boundary with expertise must be explicit

### Boundary rule

- expertise chooses who executes
- context manager helps the selected agent execute better
- context manager does not participate in authorization or routing rank

### Minimum contract

```bash
mah context find --agent <agent> --task "..."
mah context explain --agent <agent> --task "..."
mah context propose --from-session <session-ref>
mah run --with-context-memory
```

---

## D3 - Introduce a lightweight lifecycle per session/run

**Decision:** add a textual lifecycle persisted in session/provenance, without introducing a board, websocket, or mandatory daemon.

### Canonical states

```text
queued -> routed -> context-loaded -> running -> blocked -> completed
queued -> routed -> context-loaded -> running -> failed
```

### Minimum contract

```bash
mah run --trace -- "..."
mah sessions status <session-id>
mah sessions status <session-id> --json
```

### Requirements

- blocked must have an explicit reason
- routed must store a summary of the routing decision
- context-loaded must store a summary of the retrieval

### Acceptance criterion

- the operator can inspect the current state and the summarized history of an execution without opening raw logs

---

## D4 - Skill compounding will be expertise-backed, not transcript-backed

**Decision:** MAH should not copy the "compound skills" narrative naively.
It should position the loop as:

```text
session learnings -> curated context -> stronger routing confidence
```

### Implications

- `mah expertise sync` should be explained as an operational strengthening mechanism
- `mah context propose --from-session` becomes part of the compounding narrative
- proposals remain reviewed, not auto-promoted

### Minimum contract

```bash
mah expertise sync
mah context propose --from-session <session-ref>
```

---

## D5 - Bootstrap must tell the value story

**Decision:** `mah init` and `npm run setup` must explain that they prepare:

- topology
- expertise-aware routing
- context support
- visible execution

### Implications

- bootstrap docs need to be updated
- if AI-assisted bootstrap exists, it enters as an optional accelerator, not a prerequisite

### Minimum contract

```bash
mah init --yes
mah init --yes --crew bootstrap-config
```

---

## D6 - Paperclip-inspired elements are accepted only in bounded form

**Decision:** accept only three Paperclip-inspired elements:

1. **goal binding**
2. **budget awareness**
3. **governance hooks**

### Acceptable forms

- `--goal` or equivalent on runs/sessions
- cost/budget warnings and summaries
- `approval_required`, `supervision_required`, `confidential_execution` signals

### Rejected forms

- company-level budget system
- approvals engine as a separate product
- org/company operating model as the main surface

---

## 7. Proposed Architecture for `v0.9`

## Layers

1. `Policy + Topology Layer`
- continues to define the allowed set

2. `Expertise Routing Layer`
- continues to choose the best candidate among the allowed set
- gains a better explainability surface

3. `Context Manager`
- retrieves bounded operational memory after routing

4. `Execution Lifecycle Layer`
- records execution states and reasons

5. `Evidence + Proposal Loop`
- turns session outcomes into:
  - expertise signals
  - operational context proposals

## Core principle

```text
policy/topology bound
expertise decides
context prepares
runtime executes
lifecycle explains
evidence improves
```


```bash
mah context find --agent planning-lead --task "create milestones"
mah context explain --agent planning-lead --task "triage backlog"

```

## 8.3 Lifecycle

```bash
mah run --trace -- "..."
mah sessions status pi:dev:abc123
mah sessions status pi:dev:abc123 --json
```

```text
session=pi:dev:abc123
state=running
timeline=queued -> routed -> context-loaded -> running
routed_to=engineering-lead
routing_reason=matched api-auth and middleware capability profile
context=2 docs loaded, stable
last_update=2026-04-22T20:14:11Z
```

## 12. Risks

## R1 - Over-positioning

If the release speaks like Multica/Paperclip without delivering the same product type, expectation mismatch will follow.

**Mitigation:** keep repeating that MAH is an orchestration intelligence layer, not a managed platform.

## R2 - Operational verbosity

If explain/status/trace become too verbose, the operator benefit disappears.

**Mitigation:** short default output; `--json` and `--verbose` for expansion.

## R3 - Feature sprawl

Lifecycle, goals, budgets, and governance can turn into a platform backlog.

**Mitigation:** `v0.9` must stay CLI-first and reuse existing infrastructure.

## R4 - Overlapping concepts

Expertise, context, sessions, provenance, and evidence may continue to feel confusing.

**Mitigation:** docs and help must repeat the boundaries consistently.

---

## 13. RFC Approval Criteria

This RFC should be considered approved if there is agreement that:

1. `v0.9` is a **productization** release, not an architectural reinvention
2. the value axis will be **routing + context + lifecycle**
3. MAH should not compete directly as a board/app/daemon platform
4. Paperclip-inspired elements enter only in bounded form
5. future implementation can be organized into incremental milestones without breaking MAH's current model

---

## 14. Final Recommendation

The recommendation is to approve `v0.9` with the following framing:

- **Operational Agent Intelligence**
- **CLI-first**
- **runtime-agnostic**
- **policy- and topology-bounded**
- **lighter than Multica**
- **less platform-like than Paperclip**
- **more technically coherent than both for the runtime orchestration problem**

In practical terms:

- promote expertise and context to first-class public features
- add a lightweight per-session lifecycle
- reinforce the compounding loop through evidence and curation
- update bootstrap and docs so this story appears from the first use
